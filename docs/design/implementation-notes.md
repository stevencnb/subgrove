# Implementation notes

Captured during the post-fix review pass — small invariants and recovery semantics that don't fit cleanly into the algorithm descriptions but matter for understanding why the script does what it does.

- **Two-phase merge invariants.** Phase 1 (validation) only does a side effect with no `main` movement: `git -C <main_sm> fetch <linked_sm> +refs/heads/<branch>:refs/heads/<branch>`. That creates `refs/heads/<branch>` in main worktree's submodule git dir, which is harmless and gets cleaned up implicitly when the worktree is removed (or can be deleted manually). Phase 2 only runs if validation passed, so the parent FF check lives in phase 1 alone — duplicating it in phase 2 would be dead defensive code.
- **Fetch error visibility.** Local-path fetches (e.g., `git -C <main_sm> fetch <linked_sm> ...`) print noisy progress lines to stderr by default. Where the fetch's failure leads to a script `err`, we let stderr through so git's own message accompanies our wrapper. Where the fetch is best-effort and a failure is non-fatal (e.g., the optional `origin/main` mirror under `push=true`), `2>/dev/null` is acceptable.
- **`cmd_new` rollback on partial failure.** If `git submodule update --init` fails for one submodule mid-loop, an `EXIT` trap removes the half-built worktree (`rm -rf <wt>` + `git worktree prune` + `git branch -D <prefix><name>`) so a retry of the same name doesn't trip on residue. The trap is disarmed at the end of `cmd_new` so a successful run leaves everything in place.
- **`cmd_update` does not require clean state.** It only updates `refs/heads/main` in the peer's submodule git dirs (via the transient-sentinel fetch described in [update.md](update.md)) — no working-tree touch in either main worktree or the peer. Git refuses to update a checked-out branch, so a peer with `main` actually checked out gets a clear "skipped" warning rather than a silent desync. The follow-up `git submodule foreach 'git rebase main'` is documented *without* `|| true` — a real conflict should halt and prompt the user, not look like a no-op pass.
- **Peer propagation is per-submodule, never per-parent.** The parent's `refs/heads/main` is shared across all linked worktrees by git itself, so peers see new parent main automatically once `cmd_merge` runs `git merge --ff-only` in the main worktree. Only submodule git dirs are isolated and need explicit propagation.
- **End-of-run notice section.** Skips, refusals, and manual next steps are accumulated into the `ATTENTION` / `NEXT_STEPS` globals via `attention`/`next_step` and printed last by `flush_notices` under `⚠ ATTENTION` / `→ NEXT STEPS` tags, rather than only as inline `warn:` lines that scroll away in a long merge/update. Rationale and the per-command sites are in [user-data-rules.md](user-data-rules.md). `flush_notices` writes to stdout on success and is redirected (`flush_notices >&2`) on the kept-but-failed-build path; `err` carries the `✗ Error:` failure tag. `_color_on FD` gates ANSI on `NO_COLOR` being unset **and** `[[ -t FD ]]`, so the color test follows the redirected fd and captured/piped output (the whole test suite) stays plain-text greppable — assertions match the bare tag word, never an escape. macOS bash 3.2: arrays are fine and `-u` is intentionally off, so the empty-array expansions in `flush_notices` never trip.
