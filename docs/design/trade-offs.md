# Trade-offs and alternatives considered

- **Fully sharing submodule git dirs across worktrees** would have eliminated peer propagation entirely. Git provides no supported way to do this for `git submodule update --init` from a linked worktree; manual symlinking of `.git/modules/<sm>` into `.git/worktrees/<wt>/modules/<sm>` works in trivial cases but breaks per-worktree HEAD/index, since those files live inside the same git dir. Rejected — settled for *object* sharing via `--reference` (refs stay isolated, objects are borrowed) which is the supported half of the same idea.
- **Always-`push=true` on merge** would obviate `origin/main` staleness in peers but requires a network round-trip and assumes the user wants every merge published. Kept opt-in; `subgrove update` covers the deferred case.
- **`git submodule update --remote` in peers post-merge** was considered as an alternative to peer propagation, but it depends on origin being up-to-date, which is only true under `push=true`. Direct ref propagation from main worktree's submodule is push-independent and faster.
- **One-pass merge with FF checks inline** was the previous shape (and the source of a half-state risk on multi-submodule failures). Now split into validate + mutate phases; the small extra cost of the validation pass is paid only once per merge.
- **`merge -f` (force-merge that bypasses dirty checks)** was deliberately excluded. The cleanliness gate fires when uncommitted work exists in any worktree being touched; the right resolution is to commit or drop that work, not wipe it. `remove -f` exists because removal has an obvious "the user has accepted the loss" framing; merge has no equivalent moment where loss is the user's intent, so no equivalent flag. Re-adding this flag would invite the foot-gun described in [merge.md](merge.md) under "Phase 0 — discovery & filter" (stashing across a submodule SHA change → revert after merge).
- **Alternative merge strategies (squash, `--no-ff` merge commits)** are intentionally unsupported and not planned. `merge` is fast-forward-only across the parent and every touched submodule, and the FF is load-bearing rather than cosmetic: under FF a submodule's `main` lands on exactly the feat-tip SHA the parent already records as its gitlink, so parent and submodule pointers stay consistent and peer propagation ([merge.md](merge.md)) is a plain FF fetch. A non-FF strategy *inside a submodule* mints a new commit the parent's recorded gitlinks no longer match, forcing the parent merge to synthesise commits too — a cascade that buys nothing the niche needs. A parent-only `--no-ff` (submodules left FF) would stay consistent and is the natural escape hatch if a "feature X landed here" marker is ever wanted, but nothing has needed it. The suite pins the policy: every feat commit is `assert_ancestor`'d into `main` after a successful merge, which a silent slide to `--squash` would break (see [testing.md](testing.md)).
